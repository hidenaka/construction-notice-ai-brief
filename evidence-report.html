変更履歴・発注者報告は、どれくらい重い仕事か

国交省と関東地方整備局の公開資料から、工事帳票、変更、協議、履歴管理、発注者報告の背景と、サービス側で持つべきフォーマットを整理した。

結論 公式根拠 帳票 手間 サービス項目 デモ化 出典

結論

確定

正式な工事帳票である

発注者報告は、メールや雑メモではなく、工事打合せ簿、工事履行報告書、変更工程表、変更施工計画書などの正式な書類と接続する。

確定

履歴管理が求められる

国交省の情報共有システム要件では、承認状況、差し戻し、修正、差し替え、訂正、削除の履歴を残す設計になっている。

注意

平均時間は未確認

一次情報だけでは「1現場あたり平均何時間」と断定できない。ここは現場代理人への実測ヒアリングが必要。

ハッカソンでは「月◯時間削減」と断定せず、「変更・協議・報告のたびに、根拠探し、資料作成、差し戻し、承認履歴管理が発生している」と言うのが正確。

一次情報から見えた背景

確認した資料 読み取れること サービスへの意味
土木工事共通仕様書 R8.3 「報告」は、受注者が監督職員へ工事状況・結果を書面で知らせる行為。情報共有システム上の工事帳票も書面として扱われる。 発注者報告は、単なる広報成果ではなく、工事の正式なやりとりに近い位置づけで扱うべき。
土木工事共通仕様書 R8.3 重要な変更が生じた場合、受注者は変更施工計画書を、その工事に着手する前に監督職員へ提出する。 「変更が起きたら住民向け周知も更新する」だけでなく、変更理由・承認・公開日時まで残す必要がある。
土木工事共通仕様書 R8.3 工期変更では、変更日数の算出根拠、変更工程表、必要資料を添付して監督職員と協議する。 発注者向け報告には、変更前後の差分、工程影響、根拠資料リンクを含めると価値が出る。
情報共有システム機能要件 R6.3 ワークフローで、承認履歴、現在の承認状況、閲覧状況、回答希望日、受付日、回答日などを一覧確認する設計。 サービス側も「誰がいつ確認したか」「どの版を公開したか」をデータとして持つべき。
情報共有システム活用ガイドライン R6.3 工事帳票の提出・受理・承諾・指示の処理は、発議書類作成機能とワークフロー機能で扱う。 AI生成物を自動公開せず、下書き、確認、承認、公開の段階を分ける必要がある。
関東地整 スリム化ガイド調査 受注者アンケートで、資料作成の過度な要求、ローカルルール、手戻り、発注者側資料の受注者対応が確認されている。 商品名は「チラシ作成」ではなく「変更連絡・周知証跡・発注者報告支援」に寄せる。

取得できる公式フォーマット

工事打合せ簿
  • 発議者: 発注者 / 受注者
  • 発議事項: 指示 / 協議 / 通知 / 承諾 / 報告 / 提出 / その他
  • 工事名、内容、添付図、その他添付図書
  • 発注者側の処理・回答、受注者側の処理・回答
  • 年月日、監督員、現場代理人、主任技術者などの確認欄
工事履行報告書
  • 工事名、工期、日付、対象月
  • 月別の予定工程%、実施工程%、備考
  • 記事欄
  • 主任監督員、監督員、現場代理人、主任/監理技術者の確認欄
変更施工計画書・変更工程表
  • 重要な変更が生じた場合、着手前に変更事項を提出
  • 工期変更では、変更日数の算出根拠、変更工程表、必要資料を添付
  • CORINS対象工事では、変更時の登録期限も発生

手間の根拠

10% 2025年3月公表の関東地整アンケートで、役割分担上は発注者だったが受注者が対応した回答。
47% そのうち、費用を見てもらえなかった回答。受注者側の持ち出しになりうる。
6% 2026年3月公表の同アンケートで、役割分担上は発注者だったが受注者が対応した回答。
27% 2026年調査で、その費用を見てもらえなかった回答。
一次情報上の回答例 このサービスで拾うべき処理
発注者用の所内会議資料を毎月作る、第三者向け説明資料を求められる。 住民向けページと同じ元データから、発注者向けサマリーと第三者説明資料を出す。
発注者の異動時に、過去の協議経緯や関係機関協議の経緯を説明するのが負担。 変更履歴、協議メモ、公開履歴、問い合わせ対応を時系列で残す。
工事履行報告書の実施工程%の根拠を求められた。 工程変更、周知変更、作業予定、実施済み内容を紐づけて根拠欄に出す。
資料作成・修正依頼が短納期で来る、時間外連絡が多い。 テンプレート化、差分抽出、承認待ち一覧、期限管理で手戻りを減らす。

サービス側で持つべきデータ項目

現場基本工事名、発注者、施工者、連絡先、工期、作業時間帯、工種。
場所住所、緯度経度、道路名、作業帯、迂回路、駅・バス停からの徒歩影響。
規制片側交互通行、車両通行止め、歩道規制、夜間作業、騒音・振動注意。
出典道路使用許可資料、工程表、作業帯図、看板写真、発注者資料。
変更履歴変更前、変更後、理由、影響範囲、変更日時、公開日時。
承認AI抽出者、確認者、承認者、差し戻し理由、承認日時。
周知QRページ、A4、町会文面、店舗向け文面、多言語FAQ。
報告閲覧数、問い合わせ件数、変更回数、公開URL、対応ログ、未解決事項。
ここは法定書類の代替ではなく、発注者報告・近隣説明・変更連絡に使う補助資料として設計する。

ハッカソンで見せるフォーマット

画面1

既存資料の取り込み

道路使用許可資料、工程表、作業帯図、看板写真を入れる。AIは場所、期間、規制、連絡先、発注者、施工者を抽出する。

画面2

確認・承認

抽出結果に出典を紐づけ、公開可否、住民向け表現、発注者向け表現を人間が承認する。

画面3

変更履歴

夜間作業が1日延長、歩道規制が追加などを変更前後で表示し、公開履歴と承認履歴を残す。

画面4

発注者レポート

QR公開URL、A4出力、変更回数、問い合わせ件数、未解決事項、次回周知予定を1枚にまとめる。
